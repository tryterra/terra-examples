# Contributing

Notes for maintaining this repo. Customer-facing usage is in [README.md](./README.md).

## Layout

```
packages/
  cli/                       # create-tryterra-app, the scaffolder (published to npm)
    src/                     # CLI source (TypeScript, built with tsup)
    templates/               # examples, committed as static files
      health-fitness-web-app/  # generated by the mirror, do not hand-edit
```

npm workspaces. Templates live inside `packages/cli` so they're plain files, never workspace members (no dependency hoisting). Each template is a standalone, runnable project. Following create-next-app conventions, the ignore file is committed as `gitignore` (no leading dot) and restored to `.gitignore` at scaffold time, since npm strips a real `.gitignore` from published packages.

## Working on the CLI

```bash
npm install                          # root, npm workspaces
cd packages/cli
npm run dev -- ../../scratch/my-app  # run from source (tsx)
npm run build                        # bundle to dist/ (tsup); prepack runs this
```

`files: ["dist", "templates"]`. `dist/` is gitignored and produced on `prepack`; the TypeScript source is not published.

## How templates are generated

Each `templates/<name>/` is **not hand-edited here**. It's a mirror of a separate source repo (`tryterra/<name>`, e.g. `tryterra/health-fitness-web-app`), where the app is actually developed and deployed. A release-gated workflow in that source repo copies it into this repo and opens a PR. Editing `templates/*` here directly will be overwritten by the next mirror.

## Updating an existing example

1. Make the change in the example's source repo (e.g. `tryterra/health-fitness-web-app`).
2. Cut a release there to sync it:
   ```bash
   gh release create v1.2.0 --title "v1.2.0" --notes "What changed" -R tryterra/health-fitness-web-app
   ```
   (Or, without a release: `gh workflow run sync-to-examples.yml -R tryterra/health-fitness-web-app`. Plain pushes to the source `main` do not sync.)
3. That opens a **Sync <name> template** PR here. Review and merge it.
4. Publish the CLI so customers get the update (see [Publishing](#publishing-the-cli)).

The mirror is a no-op (no PR) when the template already matches the source.

## Adding a new example (new app)

Each example is one Terra product, mirrored from its own source repo. Convention: `tryterra/<product>-<form>` (e.g. `health-fitness-web-app`, `health-fitness-mobile-app`).

1. **Create the source repo** `tryterra/<product>-<form>` and build the app there as a standalone, runnable project. Commit its ignore file as `.gitignore` (the mirror renames it to `gitignore`). The repo root must have a `package.json` with a `name` and a one-line `description`: the CLI's picker and `--list` read the description, and scaffolding fails without the file (CI enforces this). If the app lives in a subdirectory, delegate the root scripts to it (see `streaming-mobile-app`). If it should offer guided setup, give the root `package.json` a `setup` script; the CLI offers to run it after install.

2. **Add the mirror workflow** to the source repo at `.github/workflows/sync-to-examples.yml` (copy it from an existing source repo). It is release-gated (`on: release: published` + `workflow_dispatch`). Point the mirror at this repo's `packages/cli/templates/<name>/`.

3. **Grant it the sync credentials.** Add `SYNC_APP_ID` and `SYNC_APP_PRIVATE_KEY` to the source repo (the `terra-examples-sync` GitHub App, already installed on this repo). Org-level secrets avoid repeating this per repo.

4. **Cut a release** in the source repo (or dispatch the workflow). It opens a **Sync <name> template** PR here that adds `packages/cli/templates/<name>/`. Review and merge it.

5. The CLI picker discovers the new template automatically (it lists `templates/` directory names). Publish a new CLI version to ship it.

## Publishing the CLI

Cut a release in **this** repo:

```bash
gh release create v1.0.0 --title "v1.0.0" --notes "What changed"
```

`.github/workflows/publish.yml` then publishes `create-tryterra-app` to npm via OIDC trusted publishing (with provenance).
